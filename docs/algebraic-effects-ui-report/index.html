代数效应如何驱动
增量 UI

当一个数从 2 变成 3，浏览器并不需要“重做整个页面”。真正的问题是： 哪些结果受影响、按什么顺序重算、最后碰哪几个 DOM 节点？

先动手试一次 代数效应是什么
A

信号是“会通知别人的值”

读取时留下联系，写入时沿着联系通知。它像电子表格的单元格，不需要手写订阅名单。

B

增量计算是“只重做变化波及的工作”

中间结果相同时，变化会被截断；连续写入可以先合并，再统一刷新。

C

代数效应是“可接管的操作接口”

代码只说“我要读”或“我要写”，根处理器统一决定记录依赖、排队、检查和刷新。

01 / THE JOURNEY

一次 UI 更新，实际上经历了什么？

先不谈任何语言理论。把一次点击拆成五站，就能看见问题的全貌。

最小更新=变了的数据+当前联系+相等判断+执行顺序

这四项少一项都可能出错：不知道联系就只能大范围重算；没有相等判断就会做无用功；顺序错了，页面可能读到一半新一半旧的数据。

03 / ALGEBRAIC EFFECTS, PLAINLY

代数效应：把“我要做什么”与“具体怎么办”分开

名字听起来像数学，其实可以先把它理解成：一组会被上层接管的操作。

组件 / 计算代码

REQUEST 请读取 count 代码不直接碰全局依赖表

像前台开出请求单，只表达意图。

→

根处理器 / handler

  • 1检查这个值是否属于当前根
  • 2先把它依赖的旧结果算新
  • 3记录“当前计算读过它”
  • 4返回最新的值
→

运行时结果

HANDLED 值 = 3 count → 当前计算 的联系已记录

一次普通读取，同时完成正确性工作。

它不只是“换一种写回调”

处理器能在一次操作周围统一加入规则：检查根是否还活着、读取前补齐结果、记录动态联系、写入后排队。这些规则不用散落在每个 signal 对象里。

类型会显示代码能做什么

只负责计算 memo 的函数允许 signal-read，却不允许 signal-write。也就是说，“算平方时偷偷改 count”不仅不推荐，而是接口本身就不放行。

同一种请求可以有不同解释

正常读取会留下依赖；untrack 临时接管读取，把它转发成“不记录联系”。API 看起来仍是读取，运行含义被局部改变。

为什么需要“待确认”这个中间状态？

count 加 2 时，“奇偶”仍是偶数。下游一开始只能知道“上游可能变了”，不能急着重跑。等奇偶值算完并发现相等，下游就可直接恢复干净。这就是 Kokaine 里 Pending 与 Stale 分开的原因。

08 / FRAMEWORK COMPARISON

React、Vue、SolidJS 与 Kokaine：同一变化，路线不同

这是对主流默认心智模型的简化比较，不是性能排名。各框架都有编译器、缓存或手动优化手段，实际结果取决于应用写法。

R

React

组件渲染 + 提交
state 更新→调用组件→比较结果→提交 DOM

默认从发生更新的组件开始重新计算其返回结果，再把必要的 DOM 改动提交出去。

本场景抽象工作范围组件子树
V

Vue

代理跟踪 + 组件更新
属性写入→触发订阅→组件更新→补丁 DOM

Proxy / ref 在运行时跟踪读写；组件本身以响应式 effect 进行渲染和更新。

本场景抽象工作范围相关组件
S

SolidJS

细粒度订阅
signal 写入→通知使用者→重算结果→改目标 DOM

运行时记录 signal 与计算之间的联系，静态 DOM 通常只建立一次。

本场景抽象工作范围绑定节点
K

Kokaine

细粒度图 + 效应处理器
write 操作→处理器排队→先结果后页面→改目标 DOM

数据路线接近 Solid；额外把“读 / 写 / UI”等能力放进 Koka 的效应类型与处理器中。

09 / FROM IDEA TO KOKA

把图上的动作翻译成 Koka 代码

这里不追求展示全部实现，只对应前面已经看见的几个关键动作。

// 原始数据：外部可以写
val count = signal(root,0)

// 中间结果：这里只允许读取 signal
val square = memo(root,0,fn(_)
  val value = count.get
  value * value
)

// 写入会由根处理器统一接管
count.modify(fn(value) value + 1)
对应图中的三件事创建数据节点运行时读取并连边写入后沿边通知
// 应用看到的能力接口
pub effect signal-read
  fun read-source(source : source<a>, tracked : bool) : a

// 根处理器给“读取”赋予运行时含义
with fun read-source(source,tracked)
  settle-source(root,source.source-id) // 先确保值是新的
  if tracked then
    track-source(root,source.source-id) // 再记录联系
  load(source.source-cell)             // 最后返回值
为什么集中处理？所有读取共享同一套正确性规则测试与替换语义更直接调用者无需接触依赖表
batch(root,fn()
  count.set(1) // 值立刻写入，暂不刷新
  count.set(2)
  count.set(3)
)
// 离开最外层 batch：按最终值统一刷新一次
处理器保管深度进入：batch depth +1写入：只排队离开最外层：flush

10 / THE BROWSER BOUNDARY

框架跑进浏览器，还要跨过一道边界

Koka 的响应式核心不依赖浏览器；DOM 解释器负责把“页面动作”翻译成真实节点操作。

为什么浏览器事件要“重新进入”处理器？

页面首次挂载结束后，Koka 的 main 已返回。用户几秒后点击按钮，是一条全新的调用栈。如果直接执行 count.set，当初包在外面的处理器已不在栈上。Kokaine 的事件跳板会先建立新的异常边界，再用 root 重新安装读写处理器，然后才运行回调。

12 / CONCLUSION

它解决了什么，又刻意没解决什么？

这套核心负责
  • 按实际读取建立细粒度数据流
  • 按正确顺序刷新，并用相等判断截断变化
  • 合并批量写入，避免重复 DOM 工作
  • 让读、写、UI 能力进入 Koka 的效应类型
  • 让条件 UI、监听器与绑定跟随所有者清理
更高层能力仍需单独设计
  • 带 key 的列表差分
  • 服务端 hydration
  • 异步资源、Suspense 与并发调度
  • 跨线程更新
  • 生产级 wasmweb DOM 适配器

增量 UI 的本质，不是“让组件自动重渲染”，而是维护一张会随执行变化的数据联系图，并在变化到来时，只把必要工作按正确顺序做完。

Kokaine 的代数效应让这张图的关键动作——读、写、提交、进入批次——从散落的内部约定，变成可处理的语言能力。
